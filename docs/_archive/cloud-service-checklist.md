# 云托管服务确认清单

> 双机 active-active 部署前，需要和云托管方确认的问题。

---

## 部署架构（优先确认）

以下问题决定了后续 Redis / TDSQL 的方案选择，需要先对齐。

1. **双活是同城还是异地？**
   同城双活和异地双活的网络延迟、故障域完全不同，直接影响锁和同步机制的设计

2. **各中心内 TDSQL 和 Redis 是否为主从部署？**
   我们的设计假设中心内有主从自动切换（秒级），需要确认是否如此，以及切换时间窗口（影响锁超时和重连策略的参数设计）

3. **跨中心的数据同步方式是什么？**
   - TDSQL：dbbridge 是异步复制还是同步复制？冲突解决策略是什么（last-writer-wins / 报错 / 合并）？
   - Redis：两个中心各自独立还是有同步机制？

4. **服务接入是否存在跨中心访问？**
   即：A 中心的用户是否可能被路由到 B 中心的服务节点？**我们的建议是用户请求按中心绑定，正常 A→A、B→B，仅当一侧整体故障时切到另一侧**。两个中心各自独立服务，不做跨中心漫游——跨中心漫游需要 Redis 状态同步和跨地域锁，复杂度远超收益

---

## Redis（各中心独立，不跨中心同步）

1. **实例是否与其他系统共用？**
   如果共用，我们需要给所有 key 加命名空间前缀，避免不同系统之间 key 冲突

2. **主从切换时，已有的 Pub/Sub 连接和 XREAD 长连接会怎样？**
   是自动重连还是直接断开？我们需要据此决定业务层的重连策略

3. **主从切换后，Lua script cache 是否会清空？**
   我们用 Lua 脚本做原子操作，如果切换后 cache 清空，需要做自动重载

4. **是否有禁用或限制的命令？**
   特别关注：`KEYS`、`SCRIPT LOAD`、`EVALSHA`、`DEBUG`

5. **单中心最大连接数和内存上限？**
   我们用连接池 + Redis Stream 缓冲事件数据，需要确认上限

---

## TDSQL（跨中心通过 dbbridge 同步）

1. **兼容的 PostgreSQL / MySQL 版本是多少？**
   影响我们的数据库驱动和 ORM 方言选择

2. **单中心最大连接数限制是多少？**

3. **默认的事务隔离级别是什么？**
   我们的并发写入行为依赖隔离级别

4. **是否支持 `SELECT 1` 做连通性检查？**
   健康检查端点需要用
