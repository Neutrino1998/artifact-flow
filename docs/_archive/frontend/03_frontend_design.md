### 1.2 左侧栏折叠

左侧栏底部提供折叠/展开按钮（`«` / `»`）：

| 状态 | 宽度 | 显示内容 |
|------|------|---------|
| 展开 | 240px | 完整操作区 + 对话历史列表 + 底部折叠按钮 `«` |
| 折叠 | 48px | 图标列：`[+]` 新建对话、`[📋]` Artifacts、`[💬]` 对话列表、底部展开按钮 `»` |

- 点击折叠按钮切换状态，带 200ms 过渡动画
- 折叠态图标 hover 显示 tooltip（"新建对话" / "Artifacts" / "对话历史"）
- 折叠态点击 `[💬]` 以 overlay 弹出对话列表浮层，选择后自动收起

### 1.3 中间区与右侧栏的拖拽分隔条

中间对话面板和右侧 Artifact 面板之间有一条可拖拽的分隔条（`┆`）：

- 拖拽时实时调整两栏宽度比例
- 鼠标悬停分隔条时光标变为 `col-resize`
- 拖拽范围约束：中间区最小宽度 400px，右侧栏最小宽度 300px
- 双击分隔条重置为默认比例

### 1.4 响应式策略

| 视口宽度 | 布局调整 |
|----------|---------|
| ≥ 1280px | 三栏完整展示 |
| 1024–1279px | 左侧栏自动折叠为图标栏（48px） |
| 768–1023px | 左侧栏隐藏（汉堡菜单唤出），Artifact 面板为覆盖式抽屉 |
| < 768px | 单栏模式，底部 Tab 切换「对话 / Artifact」|

**面板尺寸规则**：
- 左侧栏展开 240px / 折叠 48px
- 中间区与右侧栏通过拖拽分隔条调整比例，中间区最小 400px，右侧栏最小 300px
- 当 Artifact 面板不可见时（无 artifact 或用户关闭），中间区占据全部剩余宽度

## 3. 中间区：对话主面板

### 3.1 消息气泡结构

#### 用户消息

靠右对齐的气泡框，支持 Markdown 渲染：

```
                    ┌───────────────────────┐
                    │  消息内容文本（Markdown）│
                    └───────────────────────┘
                    [Copy] [Edit] [Rerun]
```

操作按钮 hover 时显示：
- **Copy**：复制消息文本
- **Edit**：编辑此消息并重新发送（创建分支，见 §3.3）
- **Rerun**：以相同内容重新发送（创建兄弟分支，见 §3.3）

#### 助手消息

不使用气泡框，内容自然铺满中间区宽度，支持 Markdown 渲染（标题、列表、表格、代码块等）。采用**分层折叠结构**，完成后只展示最终响应：

```
▶ Lead Agent (thinking)              ← 折叠
▶ 🔧 web_search (0.8s) ✓            ← 折叠
▶ Search Agent (thinking)            ← 折叠
▶ 🔧 crawl_page (2.1s) ✓            ← 折叠

最终响应内容（Markdown 全宽渲染）
# 标题
正文段落，支持 **加粗**、`代码`、链接等...
- 列表项 A
- 列表项 B

[Copy]
```

**折叠规则**（完成态）：

| UI 元素 | 默认状态 | 说明 |
|---------|---------|------|
| Thinking 内容 | 折叠 | 折叠为一行标签 `▶ Agent (thinking)`，点击展开 |
| 工具调用 | 折叠 | 折叠为一行 `▶ 🔧 tool_name (耗时) ✓/✗`，点击展开查看参数和结果 |
| Agent 切换 | 折叠 | 显示 agent 名称标签，展开查看详情 |
| 最终响应 | 展开 | Markdown 渲染的最终输出，始终可见 |
| 权限确认 | 展开 | 历史消息中显示为已确认/已拒绝的记录 |

### 3.2 SSE 工作流状态展示（流式进行中）

流式输出进行中时，消息区底部展示实时执行状态。各 SSE 事件类型对应的 UI 行为：

> SSE 事件类型定义和数据格式详见 [`streaming.md`](../../streaming.md)

#### Agent 执行指示器

**触发事件**：`agent_start`

```
┌─────────────────────────────────────────┐
│  🤖 Lead Agent                ● 执行中  │
│  ─────────────────────────────────────  │
```

显示当前执行 agent 名称和运行状态 badge。当 `agent_complete` 后有 routing 到下一个 agent 时，显示转交指示：

```
│  ✓ Lead Agent → Search Agent           │
```

#### Thinking 折叠区

**触发事件**：`llm_chunk`（当 `data.reasoning_content` 增长时）

```
│  ▼ Thinking...                         │
│  ┌─────────────────────────────────┐   │
│  │  （reasoning 内容实时更新）        │   │
│  │  用户需要了解 XX 的最新信息...     │   │
│  └─────────────────────────────────┘   │
```

- 默认折叠，显示 `▶ Thinking...` + 旋转动画
- 用户可点击展开查看实时 reasoning 内容
- 当 `reasoning_content` 停止增长、`content` 开始增长时，thinking 区自动折叠

**判断逻辑**（参考 CLI `StreamDisplay.handle_event`）：
- `reasoning_content` 有值且 `content` 为空 → 正在 thinking
- `content` 开始出现 → thinking 结束，切换到输出渲染

#### 工具调用卡片

**触发事件**：`tool_start` → `tool_complete`

进行中状态：
```
│  ┌─────────────────────────────────┐   │
│  │  🔧 web_search        ⟳ 执行中  │   │
│  │  query: "AI 最新进展"            │   │
│  └─────────────────────────────────┘   │
```

完成状态：
```
│  ┌─────────────────────────────────┐   │
│  │  🔧 web_search    ✓ 成功 (0.8s) │   │
│  │  query: "AI 最新进展"            │   │
│  │  → 3 results returned           │   │
│  └─────────────────────────────────┘   │
```

失败状态：
```
│  ┌─────────────────────────────────┐   │
│  │  🔧 crawl_page    ✗ 失败 (5.2s) │   │
│  │  url: "https://example.com"     │   │
│  │  → Error: Connection timeout    │   │
│  └─────────────────────────────────┘   │
```

卡片信息来源：
- `tool_start` 事件 → 工具名、参数摘要、loading spinner
- `tool_complete` 事件 → 成功/失败状态、耗时（`duration_ms`）、结果摘要（`result_data`）

参数显示策略：截取关键参数，超过 80 字符省略。结果摘要格式参考 CLI `StreamDisplay._format_result_data()` 的逻辑。

#### Final Response 展开区

**触发事件**：`llm_chunk`（当 `data.content` 增长时）

```
│  ─────────────────────────────────────  │
│  最终响应内容，实时打字机效果渲染...      │
│  支持 Markdown 格式 ▊                   │
```

- 使用 Markdown 实时渲染（`react-markdown`）
- 光标闪烁效果（CSS 动画）
- 自动滚动到底部

**流式结束后**（收到 `complete` 事件）：所有中间步骤（thinking、tool 卡片、agent 切换）默认折叠为一行摘要，只展示最终 response。

### 3.3 对话树与分支

依赖后端 `ConversationDetailResponse.messages` 的 `parent_id` / `children` 字段构建消息树。

> 数据结构详见 [`api.md`](../../api.md) 中 `GET /api/v1/chat/{id}` 的响应格式

#### Edit 触发分支

1. 用户点击某条已发送消息的 **Edit** 按钮
2. 该消息变为可编辑输入框，预填原始内容
3. 用户修改后提交
4. 调用 `POST /api/v1/chat`，`parent_message_id` 设为**被编辑消息的父节点 ID**
5. 后端创建新消息（新的分支），前端自动切换到新分支

#### Rerun 的语义

1. 用户点击某条消息的 **Rerun** 按钮
2. 以**完全相同的内容**重新发送
3. `parent_message_id` 设为**同一个父节点**（创建兄弟分支）
4. 后端重新执行，前端切换到新分支

#### 分支导航 UI

当某个位置存在多个兄弟消息（同一 `parent_id` 下有多个 `children`）时，在消息上方显示分支导航器：

```
            ◀  2 / 3  ▶
┌─────────────────────────────────────────┐
│  消息内容...                             │
└─────────────────────────────────────────┘
```

- 显示当前分支序号 / 总分支数
- `◀` `▶` 按钮切换分支
- 切换时，该消息及其所有子消息同步更新

**树构建逻辑**：
- 后端返回的 `messages` 是扁平数组，每条消息含 `parent_id` 和 `children` 字段
- 前端维护一条 `currentBranchPath: string[]`（从根到当前叶节点的 message ID 列表）
- 切换分支时，找到分支点，替换该点之后的路径

### 3.4 输入区

```
┌─────────────────────────────────────────────────────┐
│  ┌───────────────────────────────────┐  ┌────────┐ │
│  │  输入消息...                       │  │  发送 ↑ │ │
│  │                                   │  │        │ │
│  └───────────────────────────────────┘  └────────┘ │
│  [📎]                                              │
└─────────────────────────────────────────────────────┘
```

- **文本输入**：支持多行，`Enter` 发送，`Shift+Enter` 换行
- **发送按钮**：流式执行中变为 **Stop** 按钮（取消当前执行）
- **附件按钮** `[📎]`：预留，未来支持文件上传
- 流式执行中输入区禁用（显示 "等待响应..." 占位文本）

## 4. 右侧栏：Artifact 面板

Artifact 面板在以下时机自动打开：
- 监听到 `tool_complete` 事件中 artifact 相关工具执行成功时
- 用户从对话中点击 artifact 引用链接时

### 4.1 顶部工具栏

```
┌──────────────────────────────────────────┐
│  📄 research_report          v3 ▾       │
│  [Copy] [Download] [⟳ Refresh] [≡ List] │
└──────────────────────────────────────────┘
```

- **标题区**：当前 artifact 的 title + 版本选择器
- **Copy**：复制当前内容到剪贴板
- **Download**：下载为文件（根据 `content_type` 确定扩展名）
- **Refresh**：手动刷新当前 artifact 内容
- **List**：切换到 Artifact 列表视图（§4.4）

### 4.2 渲染模式：Markdown Preview

默认渲染模式，适用于 `content_type` 为 `markdown` 的 artifact：

```
┌──────────────────────────────────────────┐
│  [Preview] [Source] [Diff]               │
│  ────────────────────────────────────── │
│                                          │
│  # 研究报告标题                           │
│                                          │
│  ## 1. 概述                              │
│  这是一份关于 AI 最新进展的研究报告...     │
│                                          │
│  ## 2. 主要发现                           │
│  - 发现 A                                │
│  - 发现 B                                │
│                                          │
└──────────────────────────────────────────┘
```

- 使用 `react-markdown` + `remark-gfm` 渲染
- 支持表格、代码块（语法高亮）、链接
- 对于 `content_type` 为 `python`/`javascript` 等代码类型，Preview 模式显示语法高亮的代码

### 4.3 渲染模式：Source + Git Diff

#### Source 模式

显示 artifact 原始内容，带行号和语法高亮：

```
┌──────────────────────────────────────────┐
│  [Preview] [Source] [Diff]               │
│  ────────────────────────────────────── │
│   1 │ # 研究报告标题                      │
│   2 │                                    │
│   3 │ ## 1. 概述                         │
│   4 │ 这是一份关于 AI 最新进展...          │
│  ...                                     │
└──────────────────────────────────────────┘
```

#### Diff 模式

点击 **Diff** Tab 后，进入版本对比视图：

```
┌──────────────────────────────────────────┐
│  [Preview] [Source] [Diff]               │
│  对比: v2 ▾ ←→ v3 ▾                     │
│  ────────────────────────────────────── │
│   - 旧版本被删除的行                      │
│   + 新版本新增的行                        │
│     未变化的上下文行                       │
│  ...                                     │
└──────────────────────────────────────────┘
```

- 默认对比 **当前版本** vs **上一版本**
- 两个版本选择器可独立调整
- Diff 数据来源：`VersionDetailResponse.changes` 字段（`[[old_text, new_text], ...]`）
- 使用 unified diff 格式展示，高亮增删行

### 4.4 Artifact 列表视图

点击工具栏 **List** 按钮时切换到列表视图：

```
┌──────────────────────────────────────────┐
│  ≡ Artifacts (3)            [← 返回]    │

│  │  markdown · 更新于 2 分钟前        │  │
│  └────────────────────────────────────┘  │
│  ┌────────────────────────────────────┐  │
│  │  📄 data_analysis       v1   🔴   │  │
│  │  python · 更新于 5 分钟前          │  │
│  └────────────────────────────────────┘  │
│  ┌────────────────────────────────────┐  │
│  │  📄 summary             v2        │  │
```

- 每项显示 artifact ID、标题、版本号、content_type、更新时间

- 点击某项切换到该 artifact 的内容视图

### 4.5 版本导航

版本选择器位于工具栏标题旁：

```
  v3 ▾
  ┌──────────────────┐
  │ v3 (最新) - rewrite │
  │ v2 - update        │
  │ v1 - create        │
  └──────────────────┘
```

- 下拉列表显示所有版本号 + 更新类型（`update_type`）
- 选择版本后加载该版本内容
- 使用版本缓存避免重复请求

**实时更新**：
- 流式执行中，监听 `tool_complete` 事件，当 artifact 工具（`create_artifact`/`update_artifact`/`rewrite_artifact`）成功时，标记该 artifact 有更新
- Tab 上显示 🔴 更新指示
- 流结束后统一刷新有更新的 artifact（避免流式中频繁请求）
- 用户也可在流式进行中点击 Refresh 手动刷新单个 artifact

## 5. 权限确认弹窗

当 SSE 流收到 `permission_request` 事件后，随后会收到 `complete(interrupted=true)` 事件（SSE 连接关闭）。此时弹出模态对话框：

> 权限中断和恢复的完整流程详见 [`streaming.md`](../../streaming.md)

```
┌──────────────────────────────────────────┐
│          ⚠️  权限确认                     │
│  ──────────────────────────────────────  │
│                                          │
│  工具: crawl_page                        │
│  权限级别: CONFIRM                       │
│                                          │
│  参数:                                   │
│  ┌────────────────────────────────────┐  │
│  │  url: "https://example.com/page"  │  │
│  │  ...                              │  │
│  └────────────────────────────────────┘  │
│                                          │
│         [拒绝]          [确认执行]        │
└──────────────────────────────────────────┘
```

**交互流程**：
1. 弹窗出现时，SSE 连接已关闭（`complete(interrupted=true)` 已收到）
2. 弹窗为模态覆盖层，阻断其他操作
3. 展示工具名称、权限级别、调用参数
4. 用户点击「确认执行」或「拒绝」
5. 调用 `POST /api/v1/chat/{id}/resume`（`approved: true/false`）
6. 获取新的 `stream_url`，建立新的 SSE 连接
7. 新流中收到 `permission_result` → 继续后续工具执行

## 6. 状态管理概要

建议使用 Zustand 按职责划分 4 个 store：

| Store | 职责 | 关键状态 |
|-------|------|---------|
| `conversationStore` | 对话列表、当前对话、消息树、分支路径 | `conversations`, `currentConversation`, `currentBranchPath` |
| `artifactStore` | Artifact 列表、当前选中项、版本缓存、更新标记 | `artifacts`, `currentArtifact`, `versionCache`, `pendingUpdates` |
| `streamStore` | 流式执行状态、当前 agent、工具调用列表、权限请求 | `isStreaming`, `streamContent`, `reasoningContent`, `toolCalls` |
| `uiStore` | 面板折叠状态、视图模式、面板宽度 | `sidebarCollapsed`, `artifactPanelVisible`, `artifactViewMode` |

> Store 的具体字段定义和 action 签名不在本文档范围。数据结构以 [`api.md`](../../api.md) 中的 Response Schema 为准，SSE 事件格式以 [`streaming.md`](../../streaming.md) 为准。

## 附录

### A. 参考：CLI StreamDisplay 的渲染策略

CLI 端 `cli/ui.py` 中的 `StreamDisplay` 类实现了 SSE 事件到终端 UI 的映射，其核心逻辑可作为前端实现的参考：

| SSE 事件 | CLI 行为 | 前端对应 |
|----------|---------|---------|
| `agent_start` | 新建 agent Panel，重置状态 | 显示 agent badge + 运行指示器 |
| `llm_chunk` (reasoning) | 在 dim Panel 中实时更新 thinking | Thinking 折叠区实时更新 |
| `llm_chunk` (content) | 先 finalize reasoning，再在 cyan Panel 中输出 | 自动折叠 thinking，切换到 final response 区 |
| `tool_start` | finalize 当前 agent，显示 yellow Panel + 参数摘要 | 工具卡片（进行中）|
| `tool_complete` | 打印 green/red Panel（耗时 + 结果摘要） | 工具卡片更新为完成状态 |
| `agent_complete` | finalize 并打印 agent Panel | 折叠该 agent 的所有中间步骤 |
| `permission_request` | 打印 warning 提示 | 弹出权限确认弹窗 |

关键设计决策：
- **Thinking → Content 切换检测**：当 `content` 字段首次出现非空值时，说明 thinking 阶段结束
- **累积值而非增量**：`llm_chunk` 的 `content` 和 `reasoning_content` 都是累积值，前端直接替换（不是 append）
- **结果摘要格式化**：`_format_result_data()` 对不同工具类型有针对性的格式化逻辑（artifact 工具显示 message + version，搜索工具显示字符数等）
